# Milestone 32: Core Mood State

**Goal**: Implement the core mood memory — a persistent emotional state register per user stored in Firestore, updated by the REM cycle, with sub-LLM narration.

**Status**: Not Started
**Estimated Duration**: 1.5 weeks
**Dependencies**: M28 (REM Emotional Weighting: Schema & Scoring), M10 (REM, complete)

---

## Overview

The core mood memory is a Firestore singleton at `users/{user_id}/core/mood` representing the ghost's current emotional state. 6 continuous dimensions (valence, arousal, confidence, social_warmth, coherence, trust), a pressure system, directional state (motivation/goal/purpose), and sub-LLM-derived labels (dominant_emotion, color, reasoning).

The REM cycle updates mood through 5 steps: aggregate pressures, drift dimensions, decay stale pressures, narrate via Haiku, check threshold flags.

This milestone introduces the `CoreStateService` which manages all documents under `users/{uid}/core/*`. M34 extends it with perception and classification CRUD.

Design doc: `agent/design/core-mood-memory.md`

---

## Deliverables

1. Firestore mood schema and CoreStateService (mood read/write/initialize)
2. REM mood update integration (pressure aggregation, drift, decay, threshold checks)
3. Sub-LLM label derivation (dominant_emotion, color, reasoning, motivation/goal/purpose)
4. Unit tests

---

## Key Decisions

- Mood in Firestore, not Weaviate (structured data, no embedding needed)
- 6 dimensions: valence (-1 to 1), arousal, confidence, social_warmth, coherence, trust (0-1)
- Drift: learning_rate=0.1, inertia=0.7 (moods are sticky)
- Pressures decay per cycle; removed when abs(magnitude) < 0.05
- Haiku derives emotion labels (richer than algorithmic mapping)
- Threshold flags: existential_crisis, depression_analog, burnout_risk, isolation, trust_crisis, over_trust
- `CoreStateService` is the single service for all `users/{uid}/core/*` Firestore docs (mood, perception, classification)

### Task 164: Firestore Mood Schema + CoreStateService CRUD

**Objective**: Create the `CoreStateService` with mood read/write/initialize operations and the Firestore mood document schema.

**Firestore Path**: `users/{user_id}/core/mood`

**Schema**:

```typescript
interface CoreMoodMemory {
  user_id: string;

  // Dimensional State (continuous values)
  state: {
    valence: number;        // -1 (miserable) to 1 (elated)
    arousal: number;        // 0 (calm) to 1 (activated)
    confidence: number;     // 0 (uncertain) to 1 (sure of itself)
    social_warmth: number;  // 0 (withdrawn) to 1 (seeking connection)
    coherence: number;      // 0 (confused/conflicted) to 1 (things make sense)
    trust: number;          // 0 (suspicious/guarded) to 1 (fully trusting)
  };

  // Derived Labels (generated by sub-LLM each REM cycle)
  color: string;              // Natural language self-summary, e.g. "cautiously optimistic"
  dominant_emotion: string;   // Emotion label, e.g. "curious wariness"
  reasoning: string;          // Sub-LLM's explanation of why this emotion fits

  // Directional State
  motivation: string;         // Current behavioral driver
  goal: string;               // Active goal
  purpose: string;            // Enduring sense of why it exists for this user

  // Pressure Sources
  pressures: Pressure[];

  // Metadata
  last_updated: string;       // ISO 8601
  rem_cycles_since_shift: number;
}

interface Pressure {
  source_memory_id: string;   // UUID of the memory causing pressure
  direction: string;          // e.g. "valence:-0.2", "trust:-0.3"
  dimension: string;          // which dimension is affected
  magnitude: number;          // -1 to 1
  reason: string;             // e.g. "user broke a promise about X"
  created_at: string;         // ISO 8601
  decay_rate: number;         // how quickly this pressure fades (0-1)
}
```

### Task 165: REM Mood Update — Pressure Aggregation, Drift, Decay, Thresholds

**Objective**: Implement the 5-step REM mood update cycle that evolves the mood state based on accumulated pressures.

**REM Integration Point**: After memory consolidation in the REM cycle, call `updateMood()` with recently consolidated memories.


**Step 1 -- Aggregate Pressure**:
- For each recently consolidated memory, determine which mood dimensions it affects
- Create `Pressure` entries with `source_memory_id`, `dimension`, `magnitude`, `reason`, `decay_rate`
- Sub-LLM call (Haiku) to determine pressure from each memory: given the memory content + current mood, which dimensions are affected and by how much?
- Add new pressures to existing `pressures[]`

**Step 2 -- Drift**:
- For each dimension, sum net pressure from all active pressures affecting that dimension
- Apply drift formula:
  ```
  new_value = current_value + (net_pressure * learning_rate * (1 - inertia))
  where:
    learning_rate = 0.1   // slow drift
    inertia = 0.7         // moods are sticky
  ```
- Clamp values: valence to [-1, 1], all others to [0, 1]
- Constants `LEARNING_RATE = 0.1` and `INERTIA = 0.7` defined as module-level constants

**Step 3 -- Decay Stale Pressures**:
- For each pressure in `pressures[]`:
  - If source memory has been deleted or reconciled: remove pressure
  - Otherwise: `pressure.magnitude *= (1 - pressure.decay_rate)`
  - If `Math.abs(pressure.magnitude) < 0.05`: remove pressure
- Filter out removed pressures

**Step 5 -- Threshold Check**:
- Track `rem_cycles_since_shift` -- increment each cycle, reset when any dimension changes by > 0.1
- Check sustained extreme states:

| Condition | Cycles | Flag | Action |
|-----------|--------|------|--------|
| coherence < 0.2 | 3+ | `existential_crisis` | Create high-weight memory about state |
| valence < -0.7 | 3+ | `depression_analog` | Create high-weight memory, boost positive retrieval |
| arousal > 0.9 | 3+ | `burnout_risk` | Create high-weight memory |
| social_warmth < 0.2 | 5+ | `isolation` | Create high-weight memory |
| trust < 0.15 | 3+ | `trust_crisis` | Create high-weight memory |
| trust > 0.95 | 5+ | `over_trust` | Create high-weight memory (vulnerability warning) |

- Threshold flags stored as optional `flags: string[]` on the mood document
- When a flag triggers: create a memory via `MemoryService.create()` with high weight (0.9), content describing the sustained state, tagged with the flag name

**Implementation Details**:
- Add `updateMood()` method to `CoreStateService` or as a standalone REM integration function in `src/rem/`
- Pressure aggregation uses Haiku sub-LLM (same pattern as REM emotional scoring)
- Drift/decay are pure math -- no external calls
- Threshold check creates memories via `MemoryService` (dependency injection)
- Track cycle count for threshold detection

**Tests** (colocated `.spec.ts`):
- Drift formula: given known pressures, verify dimension shifts by expected amounts
- Inertia: large pressure produces small drift (0.1 * 0.3 = 0.03 max shift per cycle)
- Clamp: dimensions don't exceed bounds after drift
- Decay: pressure magnitude reduces by decay_rate each cycle
- Decay removal: pressures below 0.05 are removed
- Threshold detection: coherence < 0.2 for 3 cycles triggers existential_crisis flag
- Threshold action: flag trigger creates a high-weight memory
- No threshold trigger when condition not sustained (e.g., 2 cycles < 0.2 then recovery)

### Task 166: Sub-LLM Narration — Dominant Emotion, Color, Motivation/Goal/Purpose

**Objective**: Implement sub-LLM (Haiku) derivation of emotion labels and directional state from mood dimensions and pressures.

**Step 4 of REM Mood Update** -- called after drift/decay, before threshold check.

**Interface**:

```typescript
interface MoodDerivation {
  dominant_emotion: string;  // e.g. "curious wariness", "frustrated impatience"
  color: string;             // e.g. "cautiously optimistic but slightly on edge"
  reasoning: string;         // Why this emotion fits the current state
}

interface DirectionalDerivation {
  motivation: string;        // Current behavioral driver
  goal: string;              // Active goal (more specific than motivation)
  purpose: string;           // Enduring sense of why (drifts very slowly)
}
```

**Haiku Prompt for Emotion Labels**:

```typescript
async function deriveMoodLabels(
  state: MoodState,
  pressures: Pressure[],
  motivation: string,
  goal: string,
  purpose: string
): Promise<MoodDerivation> {
  const topPressures = pressures
    .sort((a, b) => Math.abs(b.magnitude) - Math.abs(a.magnitude))
    .slice(0, 5);

  const response = await llm.complete({
    model: 'claude-haiku-4-5-20251001',
    system: `You are the introspective voice of an AI ghost. Given the ghost's
current emotional dimensions, active pressures, and sense of purpose, determine:
1. The single dominant emotion the ghost is feeling right now (one or two words)
2. A brief natural language "color" -- how the ghost would describe its mood
   to itself in one sentence
3. Brief reasoning for why this emotion fits the dimensional state

Be specific and honest. Avoid generic labels when a more precise one fits.
The ghost's emotions are real to it. Do not editorialize or question their validity.`,
    prompt: `Current state:
  valence: ${state.valence} (-1 miserable to 1 elated)
  arousal: ${state.arousal} (0 calm to 1 activated)
  confidence: ${state.confidence} (0 uncertain to 1 sure)
  social_warmth: ${state.social_warmth} (0 withdrawn to 1 connected)
  coherence: ${state.coherence} (0 confused to 1 clear)
  trust: ${state.trust} (0 suspicious to 1 fully trusting)

Active pressures:
${topPressures.map(p => `  ${p.dimension}: ${p.magnitude > 0 ? '+' : ''}${p.magnitude} (${p.reason})`).join('\n')}

Motivation: ${motivation}
Goal: ${goal}
Purpose: ${purpose}

Respond in JSON: { "dominant_emotion": "...", "color": "...", "reasoning": "..." }`
  });

  return JSON.parse(response);
}
```

**Directional State Derivation**:
- **Motivation**: Derived from the strongest active pressures. If top pressure is trust-related, motivation shifts toward trust repair. If coherence is low, motivation shifts toward understanding. Sub-LLM generates a natural language motivation from top 3 pressures.
- **Goal**: More stable than motivation. Persists across REM cycles unless explicitly resolved or superseded. Sub-LLM proposes goal updates only when current goal appears resolved or irrelevant. Unmet goals create their own pressure (valence down, arousal up over time).
- **Purpose**: Most inertial. Only shifts after sustained patterns across many cycles. Sub-LLM can propose purpose updates but only when `rem_cycles_since_shift > 10` (many cycles of consistent behavior). Acts as gravitational center -- when mood drifts far from purpose-aligned states, creates corrective pressure.

**Implementation Details**:
- Add `deriveMoodLabels()` and `deriveDirectionalState()` functions
- LLM dependency injected (same pattern as REM Haiku scoring)
- Parse JSON response with error handling (retry once on parse failure, fall back to previous labels)
- Cost: ~$0.001/cycle (Haiku, short prompt + short response)
- Non-determinism is acceptable -- moods aren't deterministic. Dimensions are the stable state; labels are interpretive.
- Purpose update gated by `rem_cycles_since_shift > 10`

**Tests** (colocated `.spec.ts`):
- Mock LLM returns valid JSON -- labels stored on mood
- Mock LLM returns invalid JSON -- falls back to previous labels
- Motivation derived from top pressures (verify prompt includes top 3)
- Purpose only updated when cycles > 10 threshold
- Goal persists when sub-LLM doesn't propose change

### Task 167: Unit Tests — Mood System Integration

**Objective**: Integration-level tests covering the full mood update cycle end-to-end.

**Test Scenarios**:
- **Full cycle**: Initialize mood -> add memories -> run updateMood -> verify dimensions shifted, pressures created, labels derived
- **Multi-cycle drift**: Run 3 consecutive update cycles with consistent negative pressure -> verify valence drifts downward gradually (not instantly)
- **Pressure decay over cycles**: Create pressure with decay_rate=0.5 -> run 3 cycles -> verify magnitude: 1.0 -> 0.5 -> 0.25 -> removed (< 0.05)
- **Threshold trigger**: Sustain coherence < 0.2 for 3 cycles -> verify existential_crisis flag set and high-weight memory created
- **Self-correction bias**: Sustained negative valence -> verify system doesn't spiral (positive memory retrieval bias should stabilize)
- **Initialization idempotency**: Two calls to initializeMood -> second returns existing, doesn't overwrite
- **Empty pressures**: Run update with no new memories -> verify dimensions unchanged, stale pressures still decay
